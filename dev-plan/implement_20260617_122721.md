## 🎯 목표 (Goal)

검증된 자체 C++ DSP 엔진(`AuralTuneEQEngine`)을 **단일 공유 코어 `.so`**로 두고,
**비루트·전 기기**를 대상으로 두 경로로 배포한다. **HAL 글로벌(T3)은 이번 턴 보류**하되,
코어 `.so`를 그대로 포팅할 수 있도록 구조만 유지한다.

| 트랙 | 적용 대상 | 처리(두뇌) | 음질 | 조건 | 이번 턴 |
|---|---|---|---|---|---|
| **T1 · NDK `.so`** | 내부 앱(우리 앱) 재생 | 자체 `.so` | **완전 정확** | 비루트, 전 기기 | ✅ 구현 |
| **T2 · MusicFX(AudioEffect)** | **내부 앱 세션 + 외부 일부 앱**(허용 앱) | 안드로이드 표준 effect | 근사 | 비루트, 전 기기 | ✅ 구현 |
| **T3 · HAL/effect `.so` 글로벌** | 전 앱 | 자체 `.so` 시스템 등록 | 완전 정확 | 루트/Magisk | ⏸ **보류(미래)** |

핵심 원칙:
- **내부 앱은 두 경로 모두 보유** — 자체 `.so`(정밀, 기본) + MusicFX(근사)도 적용 가능.
- **외부 앱은 MusicFX(AudioEffect)만** — control-session으로 허용 앱에 근사 적용.
- **코어 `.so`는 HAL 포팅 가능하도록 OS/JNI 의존 없이 순수 유지** (T3 미래 대비).

## 📦 범위 (Scope)

- M0: 공유 코어 분리 리팩토링 (NDK 기반 + 미래 HAL 포팅 여지 확보)
- T1: 기존 AudioTrack 경로 유지 + 앱 내 플레이어(로컬 재생) 확장
- T2: 표준 `android.media.audiofx`(MusicFX 방식) 근사 — 내부 앱 세션 + 외부 일부 앱(control session)
- 근사 품질 극대화: `DynamicsProcessing` 우선 + 자체 엔진 응답곡선(freqz) 피팅
- 공유 검증: golden/scipy로 코어(T1) 정확도 보증, T2 근사 오차 정량화

## 🚫 제외 (Exclusions)

- **T3(HAL/effect `.so` 글로벌) 구현** — 이번 워크스트림 제외. 코어 구조로 **포팅 여지만 유지**.
- effect `.so`(`audio_effect.h` ABI), Magisk 모듈, SELinux 정책, `/vendor` 등록 — T3 보류에 따라 제외.
- 벤더 audio HAL 교체, AIDL effect HAL, AudioPlaybackCapture, Shizuku — 비대상.
- 신규 DSP 알고리즘 추가 — 본 워크스트림은 "배포 경로 확장"만 다룸.

## 📐 공통 규칙 (Common Rules)

1. **코어 무수정 원칙**: `AuralTuneEQEngine::process` 등 DSP 코어 로직 행위 변경 금지. 어댑터에서만 호출.
2. **코어 순수성(미래 HAL 대비)**: 코어 소스는 **JNI 의존 0**(`jni.h`/`JNIEnv`/j-타입 없음) → effect/HAL `.so`로 그대로 링크 가능해야 함. `<android/log.h>`는 **`#if defined(__ANDROID__)` 가드 + 호스트 no-op** 패턴이면 허용(현 코드가 이미 그러함 — 포팅 친화성 충족).
3. **불변식 보존**: README 13개 invariant(#1 pre-warp, #2 TDF2, #3 atomic publish, #8 NaN guard, #10 16KB page) 회귀 금지.
4. `-fno-finite-math-only`·16KB 페이지 링커 플래그 유지(`Align 0x4000`).
5. **단계 게이트**: 현재 단계 자가검증 전부 통과 전 다음 단계 진행 금지.
6. **PII 금지**: BT MAC/파일명/raw 프로파일 ID 로깅 금지.
7. **T1 vs T2 투명성**: 내부 앱 재생="정밀", 외부 앱="근사(기기 지원 기반)"임을 UI/진단에 표시.
8. 문서 범위 밖 작업 금지. 부수 이슈는 "이슈 추적"에 기록만.

## Phase 1 — T1: 내부 앱 NDK `.so` + 앱 내 플레이어 `출시 목표`

> 목적: 비루트·전 기기에서 "우리 앱으로 듣는 오디오"에 자체 `.so`로 100% 정확 적용.

### 통합 설계 (확정) — Media3 `AudioProcessor` 방식
> ExoPlayer가 디코딩·출력(AudioTrack)을 관리하고, 우리 엔진은 표준 확장점
> `androidx.media3.common.audio.AudioProcessor`로 PCM 파이프라인에 삽입한다.
> 기존 `AudioPlayerService`(자체 AudioTrack 루프)는 **TestTone 데모 전용**으로 유지.
> TestTone과 음악 재생은 **배타적**(동시 재생 금지 — 같은 engine을 두 오디오 스레드가 process하지 않도록).
```
ExoPlayer(로컬파일) → 디코드 → [AuralTuneAudioProcessor: engine.process(PCM_FLOAT)] → DefaultAudioSink → AudioTrack
```
신규 파일(:app):
- `audio/AuralTuneAudioProcessor.kt` — `BaseAudioProcessor` 상속, `onConfigure`(PCM_FLOAT/스테레오/rate↔engine 매칭), `queueInput`→`engine.process`→출력
- `audio/MusicPlayerController.kt` — ExoPlayer 빌드(커스텀 `DefaultRenderersFactory.buildAudioSink` 로 AudioProcessor 주입), play/pause/seek, 단일 engine 공유

## Phase 2 — T2: MusicFX(AudioEffect) 근사 — 내부 세션 + 외부 일부 앱 `근사`

> 목적: 안드로이드 표준 effect로 근사 적용. **내부 앱 세션**과 **외부 일부 앱**(control session) 모두.
> 처리는 OS에 위임(근사) — 순수 Kotlin/Java, 코어 `.so` 무관.

### ⚠️ 선행 PoC (착수 전 필수 게이트)
- [ ] **외부앱 커버리지 실측**: `ACTION_OPEN_AUDIO_EFFECT_CONTROL_SESSION`을 실제로 broadcast하는 앱 목록 측정. 메이저 스트리밍(Spotify/YouTube/Apple Music)은 대부분 **미지원** 예상 → "외부 일부 앱"의 실제 범위를 데이터로 확정한 뒤 Phase 진행.
- [x] PoC 도구 작성·커밋(39da55f, `feat/t2-poc-probe`): `audio/audiofx/AudioFxSessionProbe.kt` — 런타임 BroadcastReceiver(RECEIVER_EXPORTED on 33+), OPEN/CLOSE 수신 → 패키지/세션/contentType StateFlow+로그. `AuralTuneApp`에 측정 카드(실기기에서 adb 없이 감지 목록 확인). `:app:assembleDebug` SUCCESSFUL.
- [ ] ⏳ **실기기 측정 대기**(사용자): 앱 켠 채 Spotify/YouTube/로컬플레이어 재생 → 감지 건수·앱 목록 수집 → 커버리지 데이터로 본구현 진행/축소 결정.

### 신규 파일 설계 (본구현 — 게이트 통과 시) `상세 검토 2026-06-18`
| 파일(:app `audio/audiofx/`) | 책임 | 핵심 API |
|---|---|---|
| `ExternalAudioFxController.kt` | OPEN→`backend.attach(sessionId)`, CLOSE→release. 선택 프로파일 보유·생명주기(Closeable) | 런타임 receiver |
| `OsEffectBackend.kt` | sealed: `DynamicsProcessingBackend`(API28+) / `EqualizerBackend`(폴백). attach/apply/release | `DynamicsProcessing`·`Equalizer` |
| `AutoEqApprox.kt` | AutoEq 프로파일 → OS effect 밴드 게인 근사(freqz 곡선 N밴드 피팅) | — |

설계 쟁점(착수 전 결정):
- **E1 freqz 출처**: (A) `AudioEngine`에 read-only `magnitudeResponse(freqs)` JNI 추가(코어 RBJ/freqz 재사용, 정밀; `process` 무수정 원칙 무위배) **vs** (B) 필터 파라미터 직접 매핑(단순·저정밀). **권장 A**.
- **E2 폴백 한계**: `Equalizer`(API26–27)는 5밴드 고정·Q 없음 → 거친 근사. UI에 품질 저하 명시.
- **E3 권한/manifest**: `MODIFY_AUDIO_SETTINGS` 이미 선언. receiver 런타임 등록 → manifest 변경 최소.

### 구현 태스크
- [ ] OS effect 백엔드: **`DynamicsProcessing`(API 28+)** 우선, **API 26–27 또는 미지원 기기는 `Equalizer` 폴백**(거친 근사). minSdk 26 ↔ API 28 갭 분기 명시
- [ ] **근사 피팅(품질 핵심)**: 필터 1:1 매핑이 아니라 —
  - [ ] 자체 엔진으로 프로파일 최종 응답곡선(freqz) 산출(ground truth)
  - [ ] 그 곡선에 DynamicsProcessing N밴드를 최소제곱 피팅
- [ ] **내부 앱 세션 적용**: 우리 플레이어 audioSessionId에 effect attach.
  - [ ] **이중 보정 금지 정책**: 내부 재생 기본은 자체 `.so`. AudioEffect는 **동시 적용 금지(택일)** — 용도는 "외부앱용 근사를 내부에서 A/B 검증"으로 한정
- [ ] **외부 앱 적용**: `ACTION_OPEN/CLOSE_AUDIO_EFFECT_CONTROL_SESSION` 수신 → 해당 세션에 attach
- [ ] `MODIFY_AUDIO_SETTINGS` 권한, 세션 생명주기(close 시 release)
- [ ] **Play 정책 점검**: 타 앱 오디오 세션 제어의 정책 리스크 확인(접근성/오용 정책)
- [ ] 기기 미지원 graceful 처리 + UI "근사 적용(기기 지원 기반)" 표시
