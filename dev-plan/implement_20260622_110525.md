## 개발 목적

현재 AuralTune Android 앱의 그래픽 EQ, 릴리스 게이트, 회전 생명주기, AutoEQ 데이터 내재화, DB 기반 fallback, T2-OS 외부앱 근사 EQ를 하나의 릴리스 가능한 구현 흐름으로 정리한다.

| 요구사항 | 현재 상태 | 이 계획의 목표 |
|---|---|---|
| 그래픽 EQ 프리셋 저장/불러오기 | 20밴드 실시간 적용은 있음, 저장/불러오기 없음 | 현재값 자동 복원 + 이름 있는 preset CRUD |
| 릴리스 전 DEBUG 코드 revert/gate | speaker EQ, MediaStore direct play, probe UI, logs, permission 남아 있음 | release build에서 debug 코드/권한/로그 차단 |
| 회전/생명주기 안정성 | 보고서의 known rotation bug가 현재 구조에도 해당 | 오디오 스택 소유권을 retained lifecycle로 이동 |

## 공통 진행 규칙

1. 기존 워킹트리 변경은 사용자 작업으로 간주하고 revert하지 않는다.
2. release gate, lifecycle, persistence 변경은 기능 추가보다 선행한다.
3. AutoEQ profile correction과 graphic EQ manual chain은 데이터 모델과 UI에서 분리한다.
4. T1 자체 `.so` 정밀 경로와 T2 AudioEffect 근사 경로는 동시에 적용하지 않는다.
5. kill switch가 켜진 상태에서는 network fallback과 engine DSP 적용을 모두 차단한다.
6. profile명, URI, raw device key, package/session 로그는 release에서 PII/운영 로그 정책으로 gate한다.
7. 외부 upstream 데이터는 DB에 전체 HTML/raw dump로 저장하지 않고, profile 생성/적용에 필요한 최소 필드만 저장한다.
8. 각 phase 완료 전 자기검증 항목을 먼저 통과시킨다.
9. `src/main`에는 release-safe 코드만 둔다. debug-only 기능은 `src/debug` 또는 동일 FQN release no-op stub으로 분리한다.
10. DB miss fallback은 자동 타이핑 검색 루프에서 직접 네트워크를 치지 않는다.

## Phase 2 - 회전/생명주기 안정성

목적: 화면 회전 또는 Activity recreation 후 ViewModel이 닫힌 engine/player/deviceManager를 참조하지 않게 한다.

### 현재 결함

| 근거 | 내용 |
|---|---|
| `AuralTuneApp.kt` | `engine`, `deviceManager`, `musicController`가 Compose `remember` 소유 (player/tone은 삭제됨) |
| `AutoEqViewModel.kt` | 생성자에서 engine/deviceManager를 보관 |
| 보고서 §11 | 회전 시 Composable dispose가 old engine을 close하지만 ViewModel은 보존되어 dead engine 참조 |

### 권장 구조

오디오 스택 소유권은 `AudioSessionViewModel` 하나로 고정한다. `AutoEqViewModel`을 별도로 유지할 경우에도 engine/player/deviceManager를 직접 소유하지 않고 `AudioSessionViewModel`의 state/action facade만 사용한다.

```
Activity/Compose
  -> AuralTuneScreen(state + callbacks)

Retained ViewModel layer
  -> AudioSessionViewModel
     -> AudioEngine
     -> MusicPlayerController        # ExoPlayer + AuralTuneAudioProcessor (유일한 재생 경로)
     -> DeviceAutoEqManager
     -> AutoEqApi / SettingsStore facade
```

> 참고(2026-06-22 갱신): `AudioPlayerService`(AudioTrack 풀-재생)와 `TestTone`은 테스트톤 전용이라 **삭제됨**. 재생은 `MusicPlayerController`(ExoPlayer) 단일 경로. test tone 배타 정책 항목은 이번 계획에서 불필요.

### 구현 결정(2026-06-22): 단일 retained 소유자로 통합

별도 `AudioSessionViewModel` 신설 대신 **기존 `AutoEqViewModel`을 오디오 스택 단일 소유자**로 통합했다(생명주기 정합성 동일 달성, 파일/네이밍 churn 최소). 두 ViewModel이 하나의 engine을 공유하는 구조의 복잡성(상호 참조·스코프)을 피한다. 네이밍 분리는 후속 선택 사항.

### fallback 흐름

fallback은 자동 타이핑 검색 루프에서 실행하지 않는다. 로컬 DB 결과가 없을 때도 즉시 GitHub에 접속하지 않고, 아래 정책을 통과한 경우에만 동작한다.

| 조건 | 정책 |
|---|---|
| 최소 검색어 길이 | normalized query 3자 이상 |
| debounce | UI 입력 안정 후 800ms 이상 |
| 호출 방식 | 사용자가 "온라인 검색"을 누르거나 background sync job이 실행될 때 |
| single-flight | index sync는 전역 1개만 허용 |
| cooldown | 실패 후 같은 query/source에 대해 일정 시간 재시도 금지 |
| kill switch | enabled면 network fallback 전부 차단 |
| 네트워크 실패 | 기존 DB 결과 유지, UI에는 non-fatal 상태만 표시 |

```
UI query
  -> DB search
  -> miss and explicit online search/background sync allowed
  -> syncIndexFromGit() single-flight
  -> DB search retry
  -> entry found
  -> profiles DB lookup
  -> miss
  -> GitHub ParametricEQ.txt fetch
  -> parse / validate / upsert profile + filters
  -> return AutoEqProfile
```
