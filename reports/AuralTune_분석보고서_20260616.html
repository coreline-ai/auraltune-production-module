01 시스템 전역(System-wide) 적용 전략

Q. 시스템 전역 적용이 되어야 하는데(타 앱 소리까지)! 전략을 소개해줘 — 단, 자체 C++ 엔진은 반드시 재사용.

핵심 결론 “진짜 system-wide(타 앱 소리까지) ∧ 자체 엔진 100% 재사용”을 동시에 만족하는 길은 비루트에서는 양립 불가능합니다. 표준 effect는 전역이 되는 대신 처리를 OS/벤더 HAL에 위임해야 하므로 자체 엔진을 못 씁니다. 두 요구를 모두 지키는 정공법은 A. 루트/Magisk로 자체 DSP를 HAL 글로벌 이펙트로 등록하는 것이 사실상 유일합니다 (ViPER4Android · JamesDSP-루트가 쓰는 방식).

A안 상세 — 자체 엔진을 글로벌 HAL 이펙트로

기존 AuralTuneEQEngine::process()를 거의 손대지 않고 “AudioTrack 파이프라인” 대신 “Android effect HAL” 뒤에 꽂는 것이 핵심입니다.

1) 엔진을 effect 라이브러리(.so)로 래핑 — legacy C API(hardware/audio_effect.h)

EffectCreate()             // → new AuralTuneEQEngine(rate) — 세션 시작
EffectCommand(SET_CONFIG)  // → 채널/레이트/포맷 협상 (updateSampleRate 매핑)
EffectProcess(in, out)    // → engine.process(buf, frames) ★ 기존 코드 그대로
EffectCommand(SET_PARAM)   // → 컨트롤 앱이 필터값(freq/gain/Q) push
EffectRelease()            // → engine.close()
왜 적합한가 엔진이 이미 float in-place · 락-프리 · 할당-프리라 effect 콜백의 RT-safe 제약에 완벽 적합. 엔진 코어 = 무수정, 래퍼 레이어만 신규로 끝납니다.

2) HAL 등록 (Magisk 모듈)

  • .so를 /vendor/lib(64)/soundfx/에 배치
  • /vendor/etc/audio_effects.xml(또는 .conf)에 라이브러리+effect UUID를 music/글로벌 output mix postprocessing에 등록
  • Magisk가 /vendor에 오버레이 마운트(시스템 파티션 무수정) → OTA 안전

02 표준 AudioEffect를 쓰나, 자체 효과인가

Q. AudioEffect(Equalizer / BassBoost / DynamicsProcessing / LoudnessEnhancer …) 중 무엇을 쓰나? 아니면 자체 효과?

결론: 전부 자체(custom) DSP 구현 Android android.media.audiofx.* 프레임워크는 단 한 줄도 사용하지 않습니다. 검색 매칭은 전부 “EqualizerAPO”(데스크톱 프로파일 포맷명)이거나 자체 클래스명(“Loudness Equalizer”)일 뿐입니다.
나열된 표준 효과이 프로젝트
Equalizer❌ → 자체 RBJ parametric 바이쿼드 cascade
DynamicsProcessing❌ → 자체 BS.1770 오토레벨러 + soft limiter
LoudnessEnhancer❌ → 자체 ISO 226:2023 보정
BassBoost / Virtualizer / Reverb 등❌ 미사용(해당 기능 없음)
출력 경로android.media.AudioTrack만 사용(효과가 아닌 PCM 싱크)

왜 표준을 안 쓰나 — 그 차이가 핵심

표준 AudioEffect이 프로젝트(자체)
처리 위치OS/벤더 HAL (기기마다 제각각)앱 오디오 스레드 내 in-place
parametric 정확도보통 5~10밴드 고정 → AutoEq 임의 freq/Q 재현 불가임의 freq/Q peaking·shelf 1:1 재현
일관성기기별 천차만별기기 무관 동일 결과 (scipy SNR 140–157dB)
통제권앱이 계수/토폴로지 통제 불가RT 안전성·동시성까지 직접 설계
트레이드오프 자체 구현의 대가로 시스템 전역(타 앱 소리)에는 적용 불가합니다. 자체 파이프라인을 통과하는 오디오에만 적용 → 이 한계가 §1(시스템 전역 전략) 논의로 이어졌습니다. macOS 원본이 vDSP(Accelerate)로 하던 일을 Android에선 NDK C++로 이식한 셈입니다.

03 입력 → 출력까지 PCM은 어떻게 변형되나

Q. 입력에서 출력까지 어떤 방식으로 PCM 변경이 일어나나?

PCM 데이터 형식 (전 단계 동일)

  • Float32, 범위 약 [-1.0, +1.0]
  • 스테레오 인터리브: [L₀, R₀, L₁, R₁, …]
  • 한 버퍼 = 480 프레임(~10ms @ 48kHz) = 960 float

전체 흐름 — 복사는 단 1회, 이후 전부 in-place

① 음원 생성 (TestTone) FloatArray scratch[960] [JVM 힙] │ put() 복사 ← 유일한 복사 지점 ▼ ② 스테이징 Direct ByteBuffer (off-heap) [네이티브 메모리] │ engine.process(byteBuffer, 480) ▼ ③ JNI 경계 GetDirectBufferAddress → float* [zero-copy] │ ▼ ④ 네이티브 DSP 동일 버퍼를 in-place로 덮어씀 [제자리 변형] │ ▼ ⑤ 출력 track.write(byteBuffer) → HAL [같은 버퍼 그대로]

근거: AudioPlayerService.kt:176-190 — scratch→direct buffer 복사 후 in-place 처리, 같은 버퍼를 write

왜 이 구조인가 scratch(FloatArray)는 SamplesProvider.read() 인터페이스 때문에 존재. Direct ByteBuffer로 옮기는 이유는 JNI가 복사 없이 버퍼 주소를 네이티브 float*로 직접 잡기 위함.

④ 네이티브에서 일어나는 변형 — 단 3종류

(a) 스칼라 곱 — Preamp

pcm[i] *= g;   // g = preamp dB → 선형 게인

AuralTuneEQEngine.cpp:691-692

(b) 바이쿼드 차분방정식 — EQ 필터 본체 (TDF2)

yL  = b0·xL + z1L              // 출력 = 현재입력×b0 + 지연상태
z1L = b1·xL − a1·yL + z2L     // 지연상태 갱신
z2L = b2·xL − a2·yL
data[2i] = (float)yL            // 제자리 덮어쓰기

BiquadFilter.cpp:36-50

  • 입력값 + 과거 입력/출력의 가중합 → 특정 주파수 부스트/감쇠
  • z1,z2 = 지연 상태(과거 기억) — 필터가 주파수별로 다르게 반응하는 원리
  • 누적은 double, 저장만 float → 정밀도 확보(SNR 140–157dB)
  • 10개 필터 직렬 → 한 필터가 덮어쓴 버퍼를 다음 필터가 입력으로 받아 누적 보정

(c) 조건부 클램프 — Soft limiter & (d) NaN guard

// 임계 초과 샘플만 부드럽게 압축
if (|x| > 0.95) pcm[i] = copysign(0.95 + 0.05·(1−e^(−over/0.05)), x);

// 첫 프레임 비유한 → 전 필터 reset + 버퍼 제로화
if (!isfinite(pcm[0]) || !isfinite(pcm[1])) { reset all; zero buffer; }

AuralTuneEQEngine.cpp:725-737 (limiter), 743-747 (NaN guard)

변형 방식적용 단계수학
스칼라 곱Preampx · g
바이쿼드(TDF2)Manual / AutoEQ / Loudness Compy=b0·x+z1; z1=b1·x−a1·y+z2; z2=b2·x−a2·y
게인 곱(엔벨로프)Loudness Eq측정 게인을 양 채널에 곱
조건부 압축Limiter임계 초과 시 copysign(soft-knee, x)
제로화NaN guard비유한 시 버퍼=0
요약 PCM은 “한 번 복사해 네이티브 버퍼에 올린 뒤, 그 자리에서 곱셈·바이쿼드 차분방정식·조건부 클램프를 순차로 덮어쓰며 변형”됩니다. 락도 추가 할당도 없이(스냅샷 포인터 1회 로드), 입력 버퍼가 곧 출력 버퍼가 되어 AudioTrack으로 나갑니다.

04 로딩된 EQ 데이터가 통과하는 신호 체인

Q. 로딩된 EQ 데이터는 어떤 EQ를 통과해서 재생되나?

ParametricEQ.txt에서 읽은 값 = Preamp(dB 1개) + 필터 목록(최대 10개: type/freq/gainDB/Q). 이 값은 DeviceAutoEqManager.pushToEngine() → engine.updateAutoEq()로 전달되어 콜백마다 다음 순서로 처리됩니다.

입력 PCM (Float32 스테레오) ▼ ① Manual cascade ← 수동 EQ (MVP UI 미노출, 보통 비활성) ▼ ② AutoEQ Preamp ← ★ ParametricEQ.txt의 Preamp (dB→선형 게인) ▼ ③ AutoEQ cascade ← ★ ParametricEQ.txt의 필터들 (핵심) — 최대 10개 바이쿼드 직렬 ▼ ④ Loudness Compensation ← ISO 226 등청감 보정 (켤 때만, 기준 음량이면 unity) ▼ ⑤ Loudness Equalizer ← BS.1770 오토레벨러 (켤 때만) ▼ ⑥ Soft limiter ← 0.95 임계 비대칭 소프트니 ▼ ⑦ NaN guard ← 비유한 시 전체 리셋+제로화 ▼ 출력 → AudioTrack

근거: process(), AuralTuneEQEngine.cpp:677-750

즉 AutoEQ 데이터는 ②③ 단계(AutoEQ Preamp + AutoEQ cascade)를 통과합니다.

③ AutoEQ cascade — 필터가 적용되는 실제 과정

각 필터는 곧장 적용되지 않고 RBJ Audio EQ Cookbook 공식으로 바이쿼드 계수로 변환됩니다 (buildAutoEqFilter, AuralTuneEQEngine.cpp:201):

  1. 타입별 계수 생성: Peaking / LowShelf / HighShelf / HighPass → RBJ 공식
  2. Pre-warp: 프로파일 최적화 레이트 ≠ 재생 레이트면 역/순 bilinear로 중심 주파수 보정 → 레이트 무관 동일 음색
  3. Nyquist guard: 주파수 ≥ Nyquist면 실현 불가 → unity 계수로 대체(그 필터만 무력화)
  4. TDF2 처리: 스테레오 인터리브 샘플별, delay state는 오디오 스레드 소유

단계별 on/off 게이트

단계게이트비고
① ManualmanualOnMVP에선 보통 미사용
② PreampautoOn && preampOn && 게인≠1.0“Preamp” 토글 OFF면 건너뜀
③ AutoEQautoOn“Correction” 토글 = 이것
④ ISO 226 / ⑤ BS.1770loudnessCompOn / loudnessEqOn선택적
⑥ LimiterlimiterOnupdateAutoEq에서 항상 true

킬스위치 ON → autoOn/manualOn/preampOn 모두 false 강제 → 사실상 순수 패스스루(⑥ limiter만 남음).

05 7일 갱신 루틴 / 변경내역 추적 여부

Q. 7일마다 검색(갱신)하는 루틴이 있나? 변경 내역을 추적하나?

결론 주기적으로 도는 루틴 없음, 그리고 변경 내역(diff) 비교도 안 함.

① 백그라운드 주기 루틴 → 없음

함의 upstream이 11개월째 멈춘 상황에선 이 7일 갱신은 매번 동일한 INDEX.md를 통째로 다시 받는 순수 낭비이며, 변경 감지가 없어 “바뀐 게 없다”를 확인할 방법도 없습니다. → §7 번들화로 대체하면 이 문제까지 정리됩니다.

06 하이브리드(번들 + 네트워크) 전략

Q. 두 가지 방법(번들 + 네트워크 갱신)을 모두 쓰는 게 베스트인가?

결론: 하이브리드가 견고성 면에서 베스트 단, 단순 병렬이 아니라 역할을 명확히 분리해야 효과가 납니다.
번들(BUNDLED)네트워크(FETCHED)
역할신뢰의 기반(baseline) — 항상 존재보강/갱신 — 있으면 좋고 없어도 무방
보장오프라인·첫 실행 즉시 동작, SPOF 제거미래 데이터 갱신 대비 보험
실패 시(실패 없음)조용히 무시하고 번들로 폴백

단, upstream 死 상태라 네트워크 쪽엔 조건이 붙음

  1. 갱신 URL을 설정 가능(configurable)하게 — GitHub 하드코딩 대신. fork/미러/CDN으로 코드 수정 없이 출처 교체 가능
  2. 버전/세대 비교로 머지 — 번들과 네트워크 중 더 새로운 쪽 우선. 죽은 GitHub의 오래된 데이터가 새 번들을 덮어쓰는 역전 방지
  3. import는 항상 최우선·별개 — 이미 그렇게 설계됨
조회 시:  IMPORTED(사용자)  >  더 최신인 쪽(BUNDLED ↔ FETCHED 버전 비교)
첫 실행:  네트워크 불필요 (번들로 즉시 동작)
갱신:     네트워크 가능 + 출처가 번들보다 최신일 때만
현실적 권고 현재는 번들이 90%의 가치, 네트워크 갱신은 미래 보험 10%. 1순위=번들 구현(SPOF 제거), 2순위=기존 네트워크 코드를 버리지 말고 “보조”로 강등(URL 설정화 + 버전 비교만 추가).

07 정적(static) 데이터 → 번들/DB화 제안

Q. 해당 GitHub가 11개월 전 마지막 업데이트 후 정지 상태인데, 데이터를 다 가져와 DB화하면 되지 않나?

타당한 통찰 데이터가 사실상 고정이라면 현재의 “런타임 다운로드 + 캐시” 모델은 과한 설계이며, 번들/DB화가 더 합리적입니다.

왜 맞는 판단인가

  • 현재 아키텍처(TTL·백그라운드 refresh·coalescing)는 “upstream이 계속 갱신된다”는 가정의 비용 → upstream 死면 전부 무의미
  • 그럼에도 첫 실행 네트워크 의존(SPOF, 오프라인 첫 실행 불가, rate-limit/차단)은 그대로 떠안음

규모 — 번들이 충분히 현실적

필요한 건 저장소 전체가 아니라 INDEX.md + 각 기종 ParametricEQ.txt뿐(그래프 PNG·CSV 불필요).

  • 엔트리 ~5,000개 × 파일당 수백 B~2KB → 원본 ~5–10MB, 압축 시 2–3MB
  • 순수 텍스트라 압축률 높음 → APK 증가 체감 미미

구현 방식 비교

방식설명평가
A. assets 번들INDEX.md+전체 txt를 assets/autoeq/에, 기존 파서 재사용✅ 최소 변경, 기존 코드와 가장 정합
B. Room/SQLite 사전빌드 DB전처리해 .db 번들검색 빠르나 파서/모델 재작성 → 오버킬
C. 압축 JSON 1개엔트리+필터를 단일 압축 JSON중간 절충안
권장 A안 + 네트워크는 “선택적 갱신”으로 강등. 코드의 AutoEqSource.BUNDLED가 이미 이 방향을 예고하고, 파서(IndexMdParser/ParametricEqParser)는 입력이 네트워크든 로컬이든 무관해 그대로 재사용 가능.

트레이드오프

  • 빌드 파이프라인에 “스냅샷 → txt 추출 → assets 동봉” 단계 추가
  • 데이터 갱신=앱 업데이트가 되지만, upstream 死면 오히려 단점 아님
  • 라이선스: AutoEq 프로파일 MIT → 번들 OK, Jaakko Pasanen attribution 유지(이미 명시)

08 GitHub 단일 장애점(SPOF) & BUNDLED 미구현

Q. 그 GitHub가 다운 상태이면 데이터를 못 받아오나?

네 — 단일 출처에 의존 raw.githubusercontent.com/jaakkopasanen/AutoEq 딱 한 곳만 봅니다. 미러/CDN 폴백 없음.
상황GitHub 다운 시
이미 캐시된 기종 / 임포트한 프로파일✅ 정상 동작(로컬 읽기)
카탈로그가 7일 안에 받아져 캐시에 있음✅ 목록 표시(만료돼도 캐시 유지)

완충 장치: “캐시에 한 번이라도 있으면 다운그레이드하지 않음” 정책 (AutoEqRepository.kt:100-102 부근). 갱신 실패해도 기존 Loaded 유지.

발견된 갭: 번들 폴백이 설계됐으나 미구현 AutoEqSource.BUNDLED가 “shipped inside the APK as a built-in fallback” 주석과 함께 정의돼 있으나 (AutoEqModel.kt:35-40), 실제로는:
  • assets/·res/raw/에 번들 데이터 파일 0개
  • BUNDLED enum 값은 모델 정의에만 존재, 코드 어디서도 실제 생성·사용 안 됨
→ “GitHub 다운 시 내장 데이터로 폴백”이라는 의도는 있었으나 미구현 상태. README의 “first-launch download → smaller APK” 결정과 맞물려 미래 작업으로 남음.
SPOF 요약 신규 설치 + 오프라인/GitHub 다운이 겹치면 앱이 빈 카탈로그로 시작 → 사실상 사용 불가. 견고성 향상엔 ① 인기 기종 번들(BUNDLED 실구현) 또는 ② 미러 URL 추가 권장. (→ §7, §6으로 연결)

09 데이터 취득 방식 — 스트리밍이 아닌 cache-first

Q. 필요할 때마다 스트리밍으로 인터넷 통해 가져오는 건가?

정정 인터넷(HTTPS)으로 가져오는 건 맞지만 “스트리밍”이 아니라 “필요할 때 1회 다운로드 → 로컬 캐시 → 다음부터 캐시 우선(cache-first)”입니다.

“스트리밍”이 아닌 이유

  • 음악처럼 계속 흘려보내는 게 아니라 작은 텍스트 파일을 1회 통째로 받아 끝
  • 오디오 데이터가 인터넷에서 오는 게 아님. 오는 건 EQ 설정값(숫자)뿐, 실제 DSP는 전부 기기 내 로컬

네트워크가 실제 쓰이는 순간 = 딱 2번

시점받는 것이후
첫 실행 / 캐시 7일 만료INDEX.md(목록)7일 재사용, 만료돼도 백그라운드 갱신
캐시에 없는 기종 첫 선택그 기종 ParametricEQ.txtLRU 캐시(200개/5MB) 저장 → 다음엔 오프라인 OK

한 번 본 것은 오프라인 동작

  • 이미 받은 헤드폰 재선택 → 인터넷 불필요(캐시)
  • 직접 임포트 프로파일 → 영구 로컬, 인터넷 무관
  • 새 헤드폰 첫 선택 시에만 → 인터넷 필요

부가 설계: HTTPS 전용(networkSecurityConfig 강제), 동시 요청 coalescing(중복 트래픽 방지), 킬스위치(신규 네트워크 차단·캐시는 열람 가능).

10 데이터 출처 구조 — 카탈로그 / 프로파일 2단계

Q. AutoEQ는 어디서 데이터를 읽어오나? 이어셋/헤드폰별로 개별로 가지고 있나?

출처: Jaakko Pasanen의 AutoEq GitHub 저장소(온라인) 앱이 자체 데이터를 갖지 않고 두 URL에서 실시간 다운로드 (AutoEqRepository.kt:487-493) 
INDEX.md (전체 목록):
https://raw.githubusercontent.com/jaakkopasanen/AutoEq/master/results/INDEX.md

개별 프로파일 (헤드폰 하나):
https://raw.githubusercontent.com/jaakkopasanen/AutoEq/master/results/<경로>/<이름> ParametricEQ.txt

2단계 구조 — 목록 ↔ 개별

① 카탈로그 (INDEX.md)② 프로파일 (ParametricEQ.txt)
내용5,000+ 기종 목록만 (이름+측정자+경로)헤드폰 하나당 파일 하나 — 실제 EQ 계수
EQ 값 포함?❌ 없음 (이름·경로뿐)✅ Preamp + 최대 10필터(freq/gain/Q)
취득 시점첫 실행 시 1회사용자가 특정 기종 탭할 때 그 기종만
캐시7일 TTLLRU 200개/5MB

URL 조립: buildProfileUrl() (AutoEqRepository.kt:418-429)가 엔트리의 relativePath로 동적 생성(공백→%20, ( ) 유지 등 path 인코딩 보정).

답 네, 이어셋/헤드폰별로 개별 파일을 가집니다. 단 기기에 미리 들고 있는 게 아니라 “목록은 통째로 1회, EQ 값은 선택한 기종만 그때그때(lazy)” 받아오는 방식입니다. 5,000개 전체를 미리 받지 않습니다.

세 번째 출처: 사용자 임포트

GitHub에 없는 프로파일은 SAF로 ParametricEQ.txt를 직접 골라 넣을 수 있고(imp-<sha256> ID로 중복 제거), 영구 보관됩니다.

참고: “이 USB DAC엔 어떤 프로파일” 같은 기기↔프로파일 매핑은 별개 — DeviceAutoEqManager+SettingsStore가 BT MAC/USB 시리얼 키로 따로 저장.

11 프로젝트 상세 분석 — 아키텍처 · 모듈 · 버그 검증

Q. autoEQ 프로젝트 상세 분석 진행 (최초 요청)

개요

AuralTune Android — Jaakko Pasanen의 AutoEq 프로파일을 실시간 적용하는 프로덕션급 앱. macOS Swift(vDSP) 버전을 Android로 이식, 핵심 DSP를 C++/JNI로 직접 구현.

  • Kotlin 1.9.24 · C++17(NDK r27) · Jetpack Compose · AGP 8.5.2 · minSdk 26 / targetSdk 34
  • 3개 Gradle 모듈, 테스트 105 Kotlin + 6 네이티브 스위트, 16KB 페이지 호환, scipy SNR 140–157dB, TSan-clean

모듈 아키텍처

:app  (Compose UI · AudioTrack 파이프라인 · 수동 DI)
  ├─ :audio-engine   (Kotlin 래퍼 + C++ DSP 코어, JNI)   ← :app 독립, 벤더링 가능
  └─ :autoeq-data    (순수 Kotlin: 파서/캐시/리포지토리/검색)  ← :app 독립

:audio-engine·:autoeq-data는 :app 비의존 → 재사용 라이브러리 + 데모 앱 구조.

모듈별 평가

영역등급요지
:audio-engine (DSP)ART 안전성·수치 정확성·검증 모두 프로덕션급. 동시성 모델 모범적
:autoeq-dataA−정교한 코루틴 병합·방어적 파싱. 캐시 .tmp 정리만 보완
:app (UI/오디오)B단일 writer/종료 순서 설계 우수하나, 구성 변경 시 자원 소유권 모델 결함
문서/테스트A−불변식 문서·네이티브 검증 탁월. README가 코드보다 뒤처짐, 회전 테스트 부재

:audio-engine 강점

  • 동시성 모델이 핵심 자산: 매 publish마다 힙 EngineSnapshot 신규 할당 → atomic exchange → 이전 스냅샷 500ms 유예 retire. 오디오 스레드는 콜백당 포인터 1회 acquire 로드 → 진정한 lock-free·allocation-free, 중간 tearing/slot 재사용 race 구조적 불가(TSan 검증)
  • TDF2 바이쿼드 + RBJ 계수 + pre-warp + Nyquist guard, delay state 오디오 스레드 전유
  • 3중 검증(Kotlin require → JNI guard → 엔진 bounds-check), 크로스언어 상수 lockstep
README vs 코드 차이 README는 라우드니스 보정/오토레벨러를 “제외 항목”으로 기재하나, loudness/에 ISO 226:2023 + BS.1770이 완전 구현 + 테스트돼 있음. README가 코드보다 뒤처짐.

검증한 주요 발견 — 서브에이전트 “치명적 버그” 주장 정정

🔴 [실제·중요] 구성 변경(화면 회전) 시 ViewModel이 죽은 엔진을 가리킴
  • Manifest에 configChanges·screenOrientation 잠금 없음 (AndroidManifest.xml:28-37) → 회전 시 Activity 재생성
  • engine/player/deviceManager는 컴포지션 단위 remember 생성 → 회전 시 onDispose가 engine.close() 호출 (AuralTuneApp.kt:70-82, 343-347)
  • 그러나 AutoEqViewModel은 viewModel(factory=…)로 회전 후에도 보존 → 새 엔진 담은 factory 무시, 기존(닫힌 엔진 참조) 반환
결과: 회전 후 진단 폴링은 0 반환(readDiagnostics short-circuit, 크래시 없음), 그러나 토글 combine 흐름이 setAutoEqEnabled 호출 → require(handle!=0L) IllegalArgumentException (AudioEngine.kt:135) → 코루틴 크래시, 이후 토글 미반영. 오디오는 새 엔진으로 흐르나 UI 제어는 죽은 엔진에 묶임.
권장: 오디오 스택을 프로세스 스코프 싱글톤/ViewModel 소유로 이전하거나 configChanges 선언.
유형발견근거/판정
실제DisposableEffect 정리 try/finally 미보호player.close()가 throw 시 engine.close() 미실행 → 핸들 누수 (AuralTuneApp.kt:343-347 / AudioPlayerService.kt:134-137)
실제stop() 메인 스레드 runBlocking(최대 1.5s)resolver 취소 지연 시 ANR 가능 (DeviceAutoEqManager.kt:127-131)
반박“resolveJob race로 UAF (CRITICAL)”stop()이 콜백 먼저 unregister + 메인 루퍼 직렬화 → 동시 발생 불가. 과장
반박“diagnostics 폴링이 닫힌 엔진서 크래시”readDiagnostics가 handle==0L에 0 반환 → 크래시 없음
반박“ViewModel.onCleared 미구현 = 누수”엔진은 ViewModel 소유 아님(컴포지션 소유) → 안 닫는 게 옳음. 소유권 오해
한계라우트 변경 시 sampleRate 미동기화문서화된 post-MVP 항목(README 불변식 #7). 버그 아님
스멜pickFromAudioTrackHint() 죽은 코드sampleRate 동기화 비활성 후 미사용 (DeviceAutoEqManager.kt:366-369)

:autoeq-data 강점/유의

  • 파서 방어성 우수(BOM/CRLF 정규화, 64KB 상한, "NaN"/"Infinity" 명시 거부, 2중 validated)
  • caller-cancel-safe fetch 병합(LAZY + invokeOnCompletion, deferred 생명주기 바인딩) — 정교·정확, LazyDeferredBehaviorTest로 핀
  • 유의: CatalogCache는 시작 시 stale .tmp sweep 없음(경미), write↔evict 사이 protectedIds race(허용 수준)

개선 우선순위
